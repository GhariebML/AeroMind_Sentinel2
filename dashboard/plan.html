PHASE 01 Environment & Baseline Audit
COMPLETED Week 1
▼

Before writing a single line of new code, the team must fully audit the existing AeroMind codebase, verify that all 6 AirSim environments run cleanly, and establish a frozen baseline checkpoint. This phase protects the existing 91.4% MOTA from accidental regression.

// STEPS
  • 01 Clone the repo and run pytest tests/ -v --cov=src — all existing tests must pass before any changes are made.
  • 02 Run scripts/evaluate.py --scenario all and record the exact baseline: MOTA, IDF1, energy, latency. Save as results/baseline_frozen.json.
  • 03 Save the trained PPO checkpoint as checkpoints/ppo_base_frozen.pt — this is the sacred file. Tag it in git as v1.0-frozen.
  • 04 Audit the 83-dimensional state vector — map each of the 83 dimensions to its source in telemetry, confirm normalization ranges, and document in docs/state_space.md.
  • 05 Set up a CI pipeline (GitHub Actions) that automatically re-runs baseline evaluation on every PR. Any PR that drops MOTA below 80% is auto-rejected.

PHASE 02 Temporal State Encoder (GRU Layer)
COMPLETED Week 2
▼

Add a GRU-based temporal encoder that gives the system memory of the last 10 flight steps (~450ms). This module sits upstream of the existing PPO and enriches the state with historical context — critical for anticipating obstacles and predicting target movement.

// STEPS
  • 01 Create src/models/temporal_encoder.py — implement TemporalStateEncoder with a 1-layer GRU: input 83-dim, hidden 128-dim, sequence length 10.
  • 02 Implement a deque(maxlen=10) state history buffer. Handle cold-start gracefully by padding with zeros when fewer than 10 steps are available.
  • 03 The encoder outputs a 128-dim context vector. Concatenate it with the original 83-dim state to produce an enriched 211-dim vector.
  • 04 Pre-train the GRU offline on recorded AirSim trajectories (from existing evaluation runs) before connecting it to the live loop — this prevents random initialization from destabilizing the PPO.
  • 05 Write unit tests in tests/test_temporal_encoder.py: verify output shape, verify cold-start padding, verify GRU gradient flow.
✓ Deliverables
src/models/temporal_encoder.py (tested)
Pre-trained GRU weights saved to checkpoints/
Unit tests passing with >90% coverage on new module
PHASE 03 Online Adapter Network
COMPLETED Week 3
▼

Implement the OnlineAdapter — a small 2-layer MLP that sits on top of the frozen PPO and learns real-time delta corrections. The frozen base policy provides stability; the adapter provides adaptability. The adapter updates every 16 steps using a rolling replay buffer.

// STEPS
  • 01 Create src/models/online_adapter.py. Architecture: Linear(83→64) → Tanh → Linear(64→3) → Tanh. Add a learnable scalar alpha initialized to 0.1 to scale the adapter's influence.
  • 02 Implement RealTimeFlightController in src/control/rt_controller.py. It wraps both the frozen PPO and the adapter. Only adapter.parameters() are passed to the optimizer.
  • 03 Implement the rolling replay buffer as deque(maxlen=256). Every 16 steps, sample a mini-batch of 32 and run one gradient step with gradient clipping at 0.5.
  • 04 Add a --adapter-off flag to the evaluation script so you can A/B test with and without the adapter in a single command.
  • 05 Stress test: run 1000 steps in mock mode and verify alpha doesn't explode. Log adapter loss to TensorBoard under runs/adapter/.
✓ Deliverables
src/models/online_adapter.py
src/control/rt_controller.py
A/B evaluation flag --adapter-off working
TensorBoard logs confirming stable adapter training
PHASE 04 Extended Reward Function
Week 4 MED PRIORITY
▼

Extend the existing reward function with two new terms: an obstacle proximity penalty (exponential, activates within 3m) and a target proximity bonus (rewards staying in the 2–8m optimal tracking range). Both are backward-compatible with the existing reward weights.

// STEPS
  • 01 Create src/rewards/rt_reward.py extending the existing reward class. Add delta=2.0 (obstacle) and eta=1.5 (proximity) as configurable YAML parameters.
  • 02 Implement _obstacle_penalty(distances): returns 0 when min distance > 3m, then (3.0 - min_d)² as the drone approaches. Clamp max penalty at 9.0.
  • 03 Implement _proximity_bonus(target_distances): full reward for 2–8m range, 0.3 if too close, linear decay beyond 8m. Average across all visible targets.
  • 04 Plot reward component breakdown for 3 scenarios in notebooks/reward_analysis.ipynb to verify the new terms don't overwhelm the tracking signal.
  • 05 Run a 2-hour AirSim session on dense_urban with the new reward and confirm energy savings remain above 30% (must not regress from the −34.8% baseline).
✓ Deliverables
src/rewards/rt_reward.py with 5 reward components
reward_analysis.ipynb with component breakdown plots
Energy savings confirmed ≥ 30% in dense_urban
PHASE 05 Full System Integration & 22Hz Loop
Week 5 HIGH PRIORITY
▼

Wire all components together into the AeroMindRealTimeSystem main loop running at 22Hz. This phase is purely integration — no new ML code. Every component must perform within budget for the 45ms latency constraint to be met.

// STEPS
  • 01 Create src/system/aeromind_rt.py as the main entry point. It initializes: AirSim client → TemporalStateEncoder → RealTimeFlightController → RealTimeRewardComputer.
  • 02 Implement the 22Hz control loop with precise timing: use time.perf_counter() for sub-millisecond accuracy. Log actual loop frequency to confirm we're hitting 22Hz ± 1Hz.
  • 03 Profile each component's latency contribution using cProfile: detection, tracking, GRU encode, PPO forward, adapter forward, AirSim command. Total must stay under 45ms.
  • 04 Add graceful shutdown: on KeyboardInterrupt, save the adapter weights, flush logs, and land the drone safely before exit.
  • 05 Run the full integrated system for 30 minutes in all 6 AirSim scenarios. Capture a screen recording for the competition demo video.
